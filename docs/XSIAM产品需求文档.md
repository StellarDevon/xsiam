## 1. 产品概述

### 1.1 产品定位

本平台由两个独立子系统组成：

- **XSIAM（Extended Detection and Response）**：面向 SOC 的威胁检测与调查平台，整合**网络、终端、身份**三大检测域，负责数据采集、关联分析、告警聚合、事件生命周期管理，**不直接执行设备动作**。
- **SOAR（Security Orchestration, Automation and Response）**：独立的自动化编排引擎，负责**剧本编排、条件决策、设备动作调用**，通过标准接口与 XSIAM 松耦合集成，可独立部署和升级。

两者的职责边界：

| 维度 | XSIAM | SOAR |
|------|-----|------|
| 核心能力 | 检测、关联、调查、可视化 | 编排、自动化、响应执行 |
| 数据来源 | 终端/网络/身份传感器 | XSIAM 事件推送 + 内部查询 |
| 动作执行 | 展示处置工作台（记录状态） | 调用设备 Adapter 真实执行 |
| 与设备关系 | 不直接对接设备 API | 通过 Action Adapter 层对接 |
| 部署方式 | 可独立部署 | 可独立部署，独立扩容 |
| 升级影响 | 不影响 SOAR | 不影响 XSIAM 检测能力 |

### 1.2 核心价值

| 维度 | 目标 |
|------|------|
| 检测覆盖 | 覆盖网络流量、终端行为、身份认证三大信号源 |
| 噪声压降 | AI 驱动告警聚合与自动关闭，目标噪声压降率 ≥ 95% |
| 响应速度 | SOAR 自动化响应 MTTR（平均修复时间）< 60 秒 |
| 统一视图 | 统一资产、告警、事件、威胁情报视图，消除信息孤岛 |
| 编排解耦 | SOAR 独立可扩展，不因 XSIAM 升级而中断自动化流程 |
| 合规支撑 | 日志留存、响应审计、报表导出，满足等保/行业合规要求 |

### 1.3 目标用户

- **SOC 分析师（L1/L2/L3）**：日常告警处置、事件调查、手动触发剧本
- **安全管理员**：资产管理、设备接入、SOAR 剧本配置、Adapter 管理
- **安全运营负责人**：态势感知、报表分析、KPI 跟踪、自动化效果评估
- **威胁猎人**：高级威胁狩猎、日志查询、自定义检测规则

### 2.2 XSIAM ↔ SOAR 解耦设计原则

| 原则 | 说明 |
|------|------|
| **接口契约化** | XSIAM 和 SOAR 之间只通过定义好的 Webhook 事件 Schema 和 REST API 通信，不共享数据库 |
| **单向推送为主** | XSIAM 主动推送事件给 SOAR（Webhook），SOAR 按需回查 XSIAM（API）；XSIAM 不依赖 SOAR 的存在 |
| **状态最终一致** | SOAR 执行完毕后异步回写状态到 XSIAM，XSIAM 处置工作台展示最终结果 |
| **独立部署** | 两个子系统各自有独立进程/容器，可分别扩容、升级、回滚，互不影响 |
| **Adapter 隔离** | 设备接口全部封装在 SOAR 的 Adapter 层，XSIAM 无需感知任何设备协议细节 |

## 4. XSIAM 功能详细需求

### 4.1 指挥中心 / 态势总览

**功能描述：** 全局安全运营态势仪表盘，提供实时的安全指标、关键事件概览和运营效率 KPI。

#### 4.1.1 核心 KPI 指标卡

| 指标 | 说明 |
|------|------|
| 活跃事件数 | 当前未关闭事件总数，按严重级别细分 |
| 未处置告警数 | 等待分析师处置的告警总量 |
| MTTR | 平均修复时间，今日/本周趋势 |
| 告警噪声压降率 | AI 自动关闭告警占总告警的比例 |
| 资产覆盖率 | Agent 已覆盖终端 / 总终端数 |
| 严重脆弱性数 | 当前未修复的 Critical CVE 数量 |

#### 4.1.2 态势图表

- **告警趋势图（时序）：** 展示过去 24 小时/7 天告警量变化曲线，按严重级别分色
- **告警来源分布（饼图）：** 终端 / 网络 / 身份 / 邮件 占比
- **MITRE ATT&CK 热力图：** 已触发战术/技术的覆盖热力图
- **Top 受害资产：** 告警最多的终端/用户 Top 10
- **最新事件列表：** 最近活跃事件快速跳转卡片
- **地理位置攻击地图：** 可选，展示攻击源 IP 的地理分布

#### 4.1.3 仪表盘管理

- 支持自定义仪表盘布局（拖拽小组件）
- 支持按角色配置默认视图（分析师视图 / 管理视图）
- 支持时间范围筛选：今日 / 最近 7 天 / 最近 30 天 / 自定义

### 4.2 资产管理

**功能描述：** 统一管理全网资产，包括终端、用户账号、网络设备、云资产和外部服务，并持续评估其安全风险状态。

#### 4.2.1 资产发现与库存

**资产类型支持：**
- **终端：** Windows / Linux / macOS 工作站和服务器
- **移动设备：** iOS / Android（需 MDM 对接）
- **用户账号：** AD / LDAP / Azure AD / Okta / 本地账号
- **云资产：** AWS EC2/S3/RDS、Azure VM、GCP Compute
- **网络设备：** 防火墙、交换机、路由器、VPN 网关
- **外部服务：** 互联网暴露的 API、Web 服务

**库存字段（终端）：**

| 字段 | 说明 |
|------|------|
| 主机名 | 设备名称 |
| IP 地址 | 当前 IP（支持多 IP） |
| MAC 地址 | 网卡 MAC |
| 操作系统 | 类型 + 版本 + 补丁级别 |
| Agent 版本 | XSIAM Agent 版本 |
| Agent 状态 | 在线 / 离线 / 隔离 / 未安装 |
| 所属部门 | 组织架构映射 |
| 责任人 | 设备负责人 |
| 资产评分 | 0-100 综合风险评分 |
| 风险等级 | Critical / High / Medium / Low |
| 最后活跃时间 | 最后一次心跳/登录时间 |
| 关联事件数 | 当前活跃事件数 |
| 开放漏洞数 | 未修复的 CVE 数量 |

#### 4.2.2 资产风险评分

资产风险评分（0-100）综合以下因素动态计算：
- 关联的活跃告警严重级别和数量（权重 40%）
- 未修复的高危/严重漏洞数量（权重 25%）
- 资产自身敏感性（域控/数据库服务器权重更高）（权重 20%）
- 账号权限级别（管理员账号权重更高）（权重 15%）

#### 4.2.3 资产视图与筛选

- 支持多维度筛选：资产类型、风险等级、操作系统、部门、Agent 状态
- 支持关键字搜索：主机名、IP、MAC、用户名、标签
- 支持分组视图：按部门 / 风险级别 / OS 类型 / 地理位置分组
- 支持资产标签（Tag）自定义打标

#### 4.2.4 资产详情页

单资产详情页包含：
- 基础信息（主机名、IP、OS、部门、责任人）
- 安全状态摘要（风险评分、活跃告警、关联事件）
- 已安装软件列表
- 历史告警时间线
- 关联账号列表
- 网络连接快照
- 漏洞列表
- Agent 配置策略

#### 4.2.5 批量操作

- 批量修改标签/部门/责任人
- 批量执行响应动作（隔离、扫描、下发策略）
- 批量导出资产报告（CSV / PDF）

### 4.3 告警管理

**功能描述：** 汇聚来自终端、网络、身份、云、邮件等多源的安全告警，通过 AI 引擎去噪、聚合、分级，供分析师高效处置。

#### 4.3.1 告警来源

| 来源类型 | 说明 |
|----------|------|
| 终端（Endpoint） | XSIAM Agent 上报的行为检测告警 |
| 网络（Network） | 流量探针、NGFW、IDS/IPS 告警 |
| 身份（Identity） | AD/LDAP/IdP 认证异常、UEBA 告警 |
| 云（Cloud） | CloudTrail、Azure Monitor、GCP 告警 |
| 邮件（Email） | 邮件安全网关告警 |
| 第三方 Syslog | 通用 Syslog/CEF/LEEF 格式接入 |

#### 4.3.2 告警属性

| 属性 | 说明 |
|------|------|
| 告警 ID | 唯一标识（如 ALT-20250515XXXXXXXX） |
| 告警名称 | 检测规则名称 |
| 严重级别 | Critical / High / Medium / Low |
| 来源类型 | 终端 / 网络 / 身份 / 云 / 邮件 |
| 触发主机/用户 | 关联的资产或账号 |
| 关联事件 | 所属事件 ID（若已聚合） |
| MITRE 技术 | ATT&CK 战术和技术映射 |
| 检测规则 | 触发的检测规则名称和类型 |
| 处置状态 | 活跃 / 处置中 / 已关闭 / 误报 |
| 触发时间 | UTC 时间戳 |
| IOC 详情 | IP、域名、文件哈希、进程等 IOC |

#### 4.3.3 AI 智能降噪

- **自动关闭规则：** 基于历史告警的误报模式识别，自动关闭已知误报
- **告警聚合：** 相同主机/用户在时间窗口内的同类告警自动合并
- **告警去重：** 相同告警特征在滑动窗口内去重
- **噪声压降目标：** AI 自动关闭率 ≥ 95%（基于 XSIAM 基线）
- **告警计数展示：** 页面顶部实时展示今日自动关闭告警数和噪声压降率

#### 4.3.4 告警列表与筛选

列表展示字段：严重级别、告警名称、来源、主机/用户、关联事件、MITRE 技术、检测规则、处置状态、触发时间。

筛选维度：
- 严重级别（Critical / High / Medium / Low）
- 来源类型
- 处置状态
- 时间范围（24h / 7d / 30d / 自定义）
- 是否关联事件
- 指定主机/用户
- 关键字搜索（IP、哈希、进程名、用户名）

支持分组视图：按来源类型 / 严重级别 / 主机 分组显示。

#### 4.3.5 告警详情

告警详情侧滑面板包含：
- 告警基本信息（告警 ID、时间、严重级别、来源）
- 进程树（终端告警）：父进程→触发进程→子进程可视化
- IOC 分析：关联 IP / 域名 / 哈希的威胁情报判定（恶意/可疑/无害）
- 原始告警数据（JSON 格式）
- 快速操作：查看因果链、加入调查、标记误报

#### 4.3.6 告警批量操作

- 批量分配处置人
- 批量更改处置状态
- 批量关联/创建事件
- 批量运行剧本
- 批量导出

### 4.4 事件管理

**功能描述：** 将多个相关告警通过 AI 自动关联或人工合并为「事件」，作为 SOC 调查的核心工作单元，涵盖事件全生命周期管理。

#### 4.4.1 事件创建

**自动创建（AI 关联）：**
- AI 关联引擎根据以下维度自动将告警关联为事件：
  - 共同受害资产（同一主机/用户）
  - 时间窗口内的行为序列（攻击链模式）
  - MITRE ATT&CK 战术序列关联
  - 共同的 IOC（相同 C2 IP/域名/哈希）
- 事件 SmartScore（0-100）基于多维风险因子自动计算

**手动创建：**
- 分析师可手动将告警合并为新事件或加入已有事件
- 支持手动调整 SmartScore 和严重级别

#### 4.4.2 SmartScore 评分模型

SmartScore 综合以下因子计算（满分 100）：

| 评分因子 | 满分权重 | 说明 |
|----------|----------|------|
| 涉及权限级别 | 35 | 管理员/域控 = 更高得分 |
| 横向移动检测 | 28 | 发现内网渗透行为 |
| 数据外泄行为 | 22 | 异常大量数据传输 |
| 外部 C2 通信 | 13 | 与恶意 IP/域名通信 |
| 持久化机制 | +2 | 注册表自启、计划任务等 |

#### 4.4.3 事件属性

| 属性 | 说明 |
|------|------|
| 事件 ID | 唯一标识（如 INC-2025-XXXX） |
| 事件名称 | AI 自动命名或人工填写 |
| 严重级别 | Critical / High / Medium / Low |
| SmartScore | 0-100 综合威胁评分 |
| 处置状态 | 新建 / 活跃 / 调查中 / 已关闭 / 自动关闭 |
| 告警数 | 关联告警总数 |
| 受影响资产 | 关联的终端、用户、网络资产 |
| 分配人 | 负责调查的分析师 |
| MITRE 战术 | 已识别的 ATT&CK 战术列表 |
| 首次发现时间 | 最早关联告警的时间 |
| 最后更新时间 | 最近一次变更时间 |
| MTTR | 从创建到关闭的处置时长 |

#### 4.4.4 事件列表与筛选

列表展示：事件 ID、事件名称、严重级别、SmartScore、状态、告警数、分配人、MITRE 战术、首发时间、最后更新时间。

标签页快速筛选：全部 / 我的事件 / 严重事件 / 未分配 / 自动关闭 / 已解决。

筛选维度：严重级别、状态、分配人、时间范围、来源类型、关键字搜索。

#### 4.4.5 事件详情（侧滑面板）

侧滑面板包含以下子标签页：

**概览（Overview）：**
- SmartScore 评分详情与因子分解
- 受影响资产列表（含资产类型、风险级别）
- 告警链（Alert Chain）：时序展示关键告警的关联过程

**告警（Alerts）：**
- 事件内所有关联告警的详细列表
- 支持跳转告警详情

**时间线（Timeline）：**
- 从事件创建到当前的完整操作时间线
- 包含：事件创建、剧本触发、自动化动作、人工操作、状态变更

**因果关系（Causality）：**
- 跳转至因果关系视图展示攻击图谱

**MITRE ATT&CK：**
- 展示本事件中检测到的所有战术/技术映射
- 以战术分类展示技术 ID 和描述

**剧本（Playbooks）：**
- 已关联/已执行剧本列表
- 支持手动触发新剧本

**备注（Notes）：**
- 分析师可添加调查笔记
- 时间戳记录谁在何时添加了备注

#### 4.4.6 事件批量操作

- 批量分配处置人
- 批量更改状态
- 批量运行剧本
- 批量合并事件（将多个事件合并为一个）
- 批量关闭事件

### 4.5 脆弱性管理

**功能描述：** 持续扫描和追踪全网资产的安全漏洞（CVE），按业务影响优先级排序，并跟踪修复进度。

#### 4.5.1 漏洞发现

- **Agent 主动扫描：** XSIAM Agent 定期采集终端软件版本、OS 补丁状态，自动比对 CVE 数据库
- **无 Agent 网络扫描：** 支持配置定期网络扫描任务发现未安装 Agent 的资产漏洞
- **第三方扫描器集成：** 支持接入 Qualys / Tenable / Rapid7 的扫描结果

#### 4.5.2 漏洞数据

| 字段 | 说明 |
|------|------|
| CVE ID | 标准 CVE 编号 |
| CVSS 评分 | 0-10 基础评分 |
| 严重级别 | Critical(9-10) / High(7-8.9) / Medium(4-6.9) / Low(0-3.9) |
| 漏洞描述 | 简要描述和影响 |
| 影响资产数 | 受影响的终端数量 |
| 受影响软件 | 存在漏洞的软件名称和版本 |
| 修复方案 | 补丁链接或修复建议 |
| 修复状态 | 未修复 / 修复中 / 已修复 / 已接受风险 |
| 首次发现时间 | CVE 首次在本环境中被发现的时间 |
| 是否在野利用 | 是否有在野利用报告（来自威胁情报） |

#### 4.5.3 漏洞优先级排序

除 CVSS 基础评分外，系统综合以下维度计算修复优先级：
- 是否存在在野利用（Exploit In Wild）：+50% 权重
- 受影响资产的业务重要性（域控/数据库/互联网暴露资产优先）
- 受影响资产的当前活跃告警数
- 漏洞暴露时间（越久未修复优先级越高）

#### 4.5.4 脆弱性视图

- **CVE 严重级别分布图（柱状图）：** Critical / High / Medium / Low 数量
- **Top Critical CVE 列表：** 按修复优先级排序的关键漏洞
- **受影响资产趋势：** 过去 30 天受影响资产数量变化
- **修复进度看板：** 已修复 / 修复中 / 未修复 占比

#### 4.5.5 漏洞修复跟踪

- 修复状态可手动更新（修复中 / 已修复 / 接受风险）
- 支持批量标记修复状态
- 支持设置修复截止日期并发送提醒
- 修复历史记录（谁、何时、将状态改为什么）

### 4.6 威胁情报管理

**功能描述：** 管理 IOC（失陷指标）、威胁情报源，将外部威胁情报与内部事件自动关联，提升检测准确率。

#### 4.6.1 IOC 类型支持

| IOC 类型 | 示例 |
|----------|------|
| IP 地址 | 198.51.100.42 |

#### 4.6.3 情报源管理

**可集成情报源：**
- STIX/TAXII 标准接入（支持任意兼容源）
- MISP 社区情报
- VirusTotal
- 商业威胁情报平台（需配置 API Key）

**情报源管理功能：**
- 查看所有情报源接入状态（活跃/异常/已停用）
- 配置同步频率（实时 / 5分钟 / 1小时 / 每日）
- 查看每个源的 IOC 数量和最后同步时间
- 支持启用/禁用指定情报源

#### 4.6.4 IOC 搜索与管理

- 全局 IOC 搜索（IP / 域名 / 哈希）
- 按类型、判定、来源、时间筛选
- IOC 批量导入（CSV/STIX 格式）
- IOC 手动添加（单条添加）
- IOC 批量操作：修改判定、删除、导出
- IOC 关联：自动关联内网中匹配到该 IOC 的历史告警和事件

#### 4.6.5 IOC 指标规则（Indicator Rules）

- 配置 IOC 触发后的自动化动作：
  - 自动创建告警
  - 自动加入阻断列表
  - 自动触发指定剧本
- 支持配置规则的优先级和生效范围（全网 / 指定资产组）

#### 4.6.6 样本分析（威胁沙箱）

- 支持提交可疑文件（EXE/DLL/PDF/Office 文档）进行沙箱分析
- 支持提交可疑 URL 进行动态分析
- 分析结果包含：
  - 行为报告（进程创建、文件操作、网络连接）
  - 恶意性判定和置信度
  - 提取的 IOC 列表
  - MITRE ATT&CK 技术映射

#### 4.6.7 威胁响应中心（Threat Response Center）

- 展示当前全球活跃高威胁事件（如 0-Day 漏洞利用、勒索软件活动）
- 每条威胁事件包含：
  - 威胁名称和严重级别
  - 攻击者 TTPs 描述
  - 关联 IOC 数量
  - 内网受影响资产数量
  - 快速操作：调查 / 封锁 IOC / 创建检测规则

### 4.7 响应处置（Action Center）

**功能描述：** XSIAM 侧的**处置工作台**，负责动作的发起、审批流管理、状态展示和审计记录。**XSIAM 不直接调用设备 API**；所有实际设备动作由 SOAR 的 Action Adapter 层执行后异步回写状态。

> **边界说明：** XSIAM Action Center = 工作台 + 审批流 + 状态看板 + 审计日志。设备执行能力 = SOAR 的职责，见第 5 章。

#### 4.7.1 动作类型目录（展示层）

XSIAM 维护可用动作的目录，用于分析师在处置工作台发起请求。实际执行由 SOAR 完成。

**终端响应动作（目录）：**

| 动作类型 | action_type 标识 | 说明 |
|----------|-----------------|------|
| 隔离终端 | `endpoint.isolate` | 切断终端网络，仅保留 Agent 通道 |
| 解除隔离 | `endpoint.unisolate` | 恢复终端网络连接 |
| 终止进程 | `endpoint.kill_process` | 按 PID 或进程名终止进程 |
| 隔离文件 | `endpoint.quarantine_file` | 将文件移入隔离区 |
| 删除文件 | `endpoint.delete_file` | 删除指定恶意文件（需二次确认） |
| 执行脚本 | `endpoint.run_script` | 远程执行 PowerShell / Bash 脚本 |
| 实时终端 | `endpoint.live_terminal` | 建立实时远程 Shell 会话 |
| 收集取证数据 | `endpoint.collect_forensics` | 采集内存转储、事件日志等 |

**网络响应动作（目录）：**

| 动作类型 | action_type 标识 | 说明 |
|----------|-----------------|------|
| 封锁 IP | `network.block_ip` | 在防火墙封锁指定 IP / CIDR |
| 封锁域名 | `network.block_domain` | DNS 黑名单封锁域名 |
| 封锁 URL | `network.block_url` | Web 代理封锁 URL |
| 封锁哈希 | `network.block_hash` | 在所有终端封锁文件哈希 |

**身份响应动作（目录）：**

| 动作类型 | action_type 标识 | 说明 |
|----------|-----------------|------|
| 禁用账号 | `identity.disable_account` | 在 AD/IdP 禁用账号 |
| 强制重置密码 | `identity.reset_password` | 强制下次登录时改密码 |
| 撤销会话 Token | `identity.revoke_sessions` | 注销所有活跃会话 |
| 强制 MFA 认证 | `identity.enforce_mfa` | 触发二次认证挑战 |

#### 4.7.2 处置工作台（Action Center 界面）

**状态标签页：**
- **待审批**：需要人工审批的动作（显示发起人、目标资产、动作类型、关联事件、发起原因）
- **执行中**：已提交 SOAR 正在执行中的动作（含状态轮询进度）
- **今日完成**：今日所有已完成/失败动作历史（含 SOAR 回写的执行结果）
- **全部历史**：支持时间范围检索的完整动作历史

**操作流程：**
1. 分析师选择动作类型 → 选择目标资产/目标值 → 填写关联事件和原因
2. 提交后根据动作风险级别走审批流或直接提交 SOAR 执行
3. XSIAM 将动作记录写入 `actions` 集合，状态置为 `pending_approval` 或 `submitted`
4. 审批通过后通知 SOAR 触发执行；SOAR 异步回写 `running` → `completed/failed`
5. Action Center 实时展示最新状态和执行摘要

#### 4.7.3 动作审批流

| 动作风险级别 | 审批要求 | 示例动作 |
|------------|----------|----------|
| 高危 | 需上级或双人审批 | 隔离终端、禁用账号、删除文件 |
| 中危 | 发起人确认即可 | 终止进程、封锁 IP、撤销会话 |
| 低危 | 无需审批直接执行 | 收集取证、执行只读脚本 |

#### 4.7.4 脚本库（Script Library）

脚本库由 XSIAM 管理，执行时通过 SOAR `endpoint.run_script` Adapter 下发：
- 内置脚本模板（取证采集、网络快照、密码重置等）
- 支持自定义脚本上传（Python / PowerShell / Bash）
- 按 OS 类型筛选（Windows / Linux / macOS）
- 脚本版本管理（创建人、更新时间、版本说明）
- 脚本执行记录（在哪个终端、由谁、何时、执行结果）

#### 4.7.5 动作审计日志

所有动作均生成不可篡改的审计记录：
- 动作类型、目标资产/值、发起人、关联事件
- 发起时间 / 审批时间 / 提交 SOAR 时间 / 完成时间（UTC）
- 审批人（如适用）
- SOAR 回写的执行结果（成功/失败/取消，含错误详情）
- 执行所用的 Adapter 类型和版本

### 4.8 日志管理 / 查询中心

#### 4.8.2 XQL 查询语言

支持类 SQL 的 XQL（Extended Query Language）查询语法：

**基本结构：**
```
dataset = <数据集名称>
| filter <条件>
| fields <字段列表>
| sort desc <排序字段>
| limit <返回行数>
```

**支持的 XQL 操作符：**

| 操作符 | 说明 |
|--------|------|
| `filter` | 条件过滤 |
| `fields` | 字段选择 |
| `sort` | 排序（asc/desc） |
| `limit` | 结果行数限制 |
| `dedup` | 按字段去重 |
| `comp` | 聚合计算 |
| `join` | 多数据集关联 |
| `union` | 数据集合并 |
| `alter` | 字段计算/转换 |
| `arrayexpand` | 数组展开 |
| `tstats` | 时序统计 |

**支持的函数：** count()、sum()、avg()、min()、max()、to_string()、to_epoch()、extract()、json_extract()、array_length()、if()、coalesce()

#### 4.8.3 查询中心 UI

**布局：**
- 左侧面板：已保存查询列表 + 数据集浏览器（字段展开）
- 中央：XQL 编辑器（支持语法高亮、Tab 补全）+ 查询结果表
- 右侧面板：XQL 语言参考（操作符、函数、操作符说明）

### 4.10 终端安全

**功能描述：** 基于 XSIAM Agent 的终端行为监控和威胁检测，覆盖进程、文件、注册表、网络连接、内存等关键维度。

#### 4.10.1 终端行为监控

**进程监控：**
- 记录进程创建/终止事件（进程名、路径、哈希、父进程、命令行参数、用户上下文）
- 检测：
  - Living-off-the-Land（LOL）滥用系统工具
  - PowerShell 编码命令执行
  - 可疑的进程空洞注入（Process Hollowing）
  - 异常的子进程关系（如 Word 生成 PowerShell）
  - Mimikatz / 凭证转储工具特征

**文件监控：**
- 记录文件创建/修改/删除/重命名事件
- 检测：
  - 勒索软件加密行为（大量文件在短时间内被修改/重命名）
  - 危险路径写入（System32、临时目录）
  - 可执行文件从异常位置运行

**注册表监控：**
- 记录注册表键创建/修改/删除事件
- 检测：
  - 自启动注册表项（Run/RunOnce）修改
  - 安全工具禁用（如 Defender 相关键值）

**网络连接监控（终端侧）：**
- 记录终端发起/接受的网络连接（远端 IP、端口、协议、进程）
- 检测：
  - 与已知 C2 IP/域名通信
  - DNS over HTTPS 隧道
  - 异常的大量数据外传

**内存保护：**
- 防止 LSASS 内存读取（凭证转储防护）
- 防止进程注入

#### 4.10.2 终端检测规则

- **BIOC（Behavioral Indicator of Compromise）：** 基于行为序列的检测规则
- **IOC 匹配：** 自动将终端采集的 IP/域名/哈希与威胁情报实时匹配
- **ML 异常检测：** 基于基线的进程行为异常检测

### 4.11 网络安全检测

**功能描述：** 分析网络流量、防火墙日志、DNS 请求，识别网络层面的威胁行为，包括横向移动、数据外泄、C2 通信、扫描探测等。

#### 4.11.1 网络流量分析

**检测场景：**
- **数据外泄检测：** 单次传输超过阈值的异常大量数据上传（可配置阈值，默认 100MB）
- **横向移动：** 内网主机间异常的端口扫描、SMB 爆破、PsExec 等
- **C2 通信：** 与已知恶意 IP/域名通信、信标（Beaconing）周期性通信
- **端口扫描：** 短时间内对多 IP/端口的扫描行为
- **VPN 暴力破解：** VPN 网关异常高频认证失败

#### 4.11.2 DNS 分析

- 检测 DGA（域名生成算法）域名访问
- 检测 DNS 隧道（通过 TXT 记录外泄数据）
- 检测异常 DNS 查询量（单主机短时间大量 NXDOMAIN）
- 支持 DNS 日志接入和查询

#### 4.11.3 网络资产感知

- 通过流量分析自动发现内网资产（无 Agent 设备）
- 识别内网服务（Web 服务、数据库、文件共享、RDP）
- 识别未授权设备（新接入内网的未知设备）

### 4.12 身份安全

**功能描述：** 基于 UEBA（用户和实体行为分析），监控用户认证行为和权限使用，识别账号攻击、内部威胁和身份滥用。

#### 4.12.1 身份数据源

- Active Directory / LDAP
- Azure Active Directory
- Okta / OneLogin / Ping Identity
- RADIUS / VPN 认证日志
- 应用程序登录日志（SSO）

#### 4.12.2 身份风险评分

每个用户账号持续计算风险评分（0-100），综合以下维度：
- 登录地理位置异常（新国家/新城市）
- 登录时间异常（非工作时间、非正常工作日）
- 设备异常（新设备、未知设备）
- 认证失败频率
- 权限使用异常（首次访问敏感资源）
- 关联告警和事件数量

#### 4.12.3 身份威胁检测

| 检测场景 | 说明 |
|----------|------|
| 凭证填充攻击 | 多 IP 对同一账号的高频登录尝试 |
| 密码喷洒攻击 | 同一 IP 对多账号的低频登录尝试 |
| 不可能旅行（Impossible Travel） | 同一账号在不可能的时间内从两个地理位置登录 |
| 首次境外登录 | 账号首次从新国家/地区登录 |
| DCSync 攻击 | 非域控主机发起的域复制请求 |
| 黄金票据（Golden Ticket） | 异常的 Kerberos 票据使用模式 |
| 权限提升 | 普通用户突然获得管理员权限 |
| 大量文件访问 | 短时间内访问异常数量的文件/共享 |
| 服务账号异常 | 服务账号从异常主机登录或执行交互式操作 |

#### 4.12.4 特权账号管理（PAM 可视化）

- 识别和展示环境中的特权账号（Domain Admin、本地 Admin、服务账号）
- 追踪特权账号的使用行为
- 标记长期未使用的特权账号（僵尸账号）

#### 4.12.5 用户详情页

单用户详情页包含：
- 基础信息（用户名、邮箱、部门、职级、账号类型）
- 风险评分和风险因子分解

- 关联事件列表

### 4.15 AI 智能助手（Copilot）

**功能描述：** 内置的 AI 安全助手（Copilot），支持自然语言问答、事件深度分析摘要、查询辅助和上下文驱动的处置建议。Copilot 是分析师与平台数据之间的智能交互层，不替代分析师判断，而是压缩信息收集和理解的时间成本。

#### 4.15.1 对话式问答

- 自然语言询问平台数据："最近 24 小时有哪些 Critical 告警未分配？"
- 事件分析：输入事件 ID，AI 自动生成结构化事件分析摘要，包含：
  - 攻击链推断（初始访问 → 横向移动 → 目标行为）
  - SmartScore 高分因子解释
  - 受影响资产风险等级
  - 建议处置步骤（优先级排序）
- 威胁解释：解释 MITRE ATT&CK 技术编号、IOC 含义、CVE 漏洞危害
- 对话上下文保持：Copilot 在单次会话内记住前序问答，支持追问

#### 4.15.2 自然语言转 XQL（NL2XQL）

- 分析师用自然语言描述查询意图，Copilot 生成可执行 XQL 语句
- 示例：
  - 输入："过去 7 天哪些主机向境外 IP 上传了超过 1GB 的数据？"
  - 输出：完整 XQL 语句 + 字段说明
- 生成的 XQL 可直接在查询中心编辑和执行，不自动执行
- 支持对生成结果追问修改："只看 Windows 主机"→ 重新生成
- 查询历史推荐：基于当前事件上下文推荐相关的常用查询模板

#### 4.15.3 告警与事件 AI 摘要

- **告警摘要**：高严重级别告警自动生成一句话摘要，展示在告警列表行内（可关闭）
- **事件 AI 摘要**：事件详情页顶部提供 AI 生成的事件概述卡片，内容包括：
  - 事件背景（攻击场景判断）
  - 关键证据（最高置信度的 IOC、异常行为、受影响账号）
  - 已识别的 MITRE 战术路径
  - 推荐的下一步调查动作
- 摘要生成由分析师手动触发或在事件创建后 30 秒内自动触发（可配置）
- 分析师可对摘要质量打分（👍/👎），反馈用于模型调优

#### 4.15.4 误报识别建议

- Copilot 分析历史已标记为"误报"的告警，识别共同特征
- 当新告警与误报模式高度相似时，在告警列表展示"疑似误报"标识和相似度
- 分析师确认误报后，Copilot 将该模式加入个人/团队误报知识库

#### 4.15.5 Copilot 入口与交互

- **全局侧边栏**：任意页面均可展开 Copilot 侧边栏，保持上下文感知
- **事件详情内嵌**：事件详情页内嵌 Copilot 面板，自动注入当前事件 ID 为上下文
- **查询中心辅助**：XQL 编辑器右侧嵌入 NL2XQL 输入框
- **快速建议 Chips**：Copilot 侧边栏顶部展示上下文相关的快速操作建议：
  - "分析 INC-2026-0042 的攻击链"
  - "为什么这个事件 SmartScore 是 87？"
  - "列出所有未分配的 Critical 事件"
  - "生成最近一周的 SOC 运营摘要"

### 4.16 因果关联引擎（CAE）

**功能描述：** 因果关联引擎（Causality Analysis Engine）是 XSIAM 的核心后端计算模块，负责将来自终端、网络、身份多个域的原始事件自动关联为有意义的攻击故事，驱动告警→事件的自动聚合，以及攻击链图谱的数据生成。CAE 是告警降噪和事件质量的基础保障。

> **与 4.14 的关系**：4.14（因果关系分析/攻击链视图）是前端可视化展示层，CAE 是其后端数据生成引擎。没有 CAE，攻击链图谱只能手动拼接。

#### 4.16.1 关联维度

CAE 从以下维度建立事件之间的因果关系：

| 关联维度 | 说明 | 示例 |
|----------|------|------|
| **进程谱系** | 父子进程关系链，识别恶意进程的创建路径 | Word.exe → powershell.exe → mshta.exe |
| **资产共性** | 相同主机或用户账号在时间窗口内触发的相关告警 | 同一主机在 10 分钟内触发 3 个不同规则的告警 |
| **IOC 共性** | 共同引用同一个恶意 IP/域名/文件哈希的告警 | 多台主机访问同一 C2 IP |
| **行为序列** | 符合 MITRE ATT&CK 战术推进顺序的告警序列 | T1566（钓鱼）→ T1059（脚本执行）→ T1041（数据外泄） |
| **横向移动路径** | 同一攻击者从一台主机跳转到另一台主机的行为链 | 源主机 SMB 认证失败 → 目标主机新进程出现 |
| **身份关联** | 同一用户账号在不同系统中的异常行为关联 | Okta 异常登录 → AD 权限变更 → 文件批量访问 |

#### 4.16.2 关联算法要求

- **时间窗口**：可配置关联时间窗口（默认 24 小时，可设置 1h/6h/24h/72h）
- **置信度评分**：每条关联关系输出置信度（0-100），低于阈值的关联不自动归入事件
- **增量计算**：新告警到来时增量更新已有因果图，不需全量重算
- **多根因支持**：一个事件可能有多个独立攻击入口，支持多根因节点
- **去环检测**：因果图为有向无环图（DAG），引擎自动检测并拒绝循环关联
- **最大图规模**：单事件因果图节点上限 500，超出时按置信度剪枝

#### 4.16.3 自动告警→事件聚合规则

CAE 计算出的关联关系触发以下自动聚合逻辑：

| 规则 | 条件 | 动作 |
|------|------|------|
| **新建事件** | ≥2 个告警建立因果关联，且不属于任何现有事件 | 自动创建新事件，命名由 AI 生成 |
| **加入已有事件** | 新告警与已有事件的告警存在因果关联 | 自动将新告警加入该事件 |
| **事件合并** | 两个独立事件的告警集合之间新建了因果关联 | 提示分析师合并（不自动合并，需人工确认） |
| **孤立告警** | 告警在 72 小时内未建立任何因果关联 | 保持独立告警状态，由分析师人工处理 |

#### 4.16.4 因果图数据模型

CAE 输出的因果图存储为图结构，节点和边定义如下：

**节点类型：**

| 类型 | 字段 | 说明 |
|------|------|------|
| Process | pid, name, path, cmdline, sha256, user | 操作系统进程 |
| File | path, sha256, operation | 文件操作 |
| Network | src_ip, dst_ip, dst_port, protocol, bytes | 网络连接 |
| Registry | key, value, operation | 注册表操作 |
| Alert | alert_id, severity, rule_name | 告警节点（标记到触发告警的具体事件节点上） |
| User | username, domain, account_type | 用户账号 |
| Asset | asset_id, hostname | 主机资产 |

**边类型（因果关系）：**

| 边类型 | 说明 |
|--------|------|
| `spawned` | 进程创建子进程 |
| `wrote_file` | 进程写入文件 |
| `executed_file` | 进程执行文件 |
| `connected_to` | 进程发起网络连接 |
| `modified_registry` | 进程修改注册表 |
| `logged_in_as` | 进程以指定用户身份运行 |
| `lateral_move_to` | 从一台主机横向移动到另一台 |
| `triggered_alert` | 节点触发了告警 |

#### 4.16.5 配置与管理

- **关联规则配置**：安全管理员可启用/禁用特定关联维度，防止误关联
- **阈值配置**：配置自动聚合所需的最小关联置信度（默认 70）
- **时间窗口配置**：按资产类型配置不同时间窗口（域控 72h，普通终端 24h）
- **引擎健康监控**：实时展示 CAE 处理队列深度、平均关联延迟、图节点总量

### 4.17 SmartScore 评分引擎

**功能描述：** SmartScore 是对事件综合威胁程度的实时动态评分（0-100），是分析师判断事件优先级的核心依据。评分引擎在事件生命周期内持续更新，每次新告警加入、资产信息变化、威胁情报更新时均触发重新计算。

#### 4.17.1 评分架构

SmartScore 由四个维度的子分数加权合成，各维度内部再细分评分因子：

```
SmartScore（0-100）
├── 攻击影响维度（Impact）       权重 40%
│   ├── 受影响资产业务重要性
│   ├── 受影响账号权限级别
│   └── 受影响资产数量
│
├── 攻击行为维度（Behavior）      权重 35%
│   ├── 已识别 MITRE 战术阶段数
│   ├── 横向移动检测
│   ├── 数据外泄行为
│   ├── 持久化机制
│   └── C2 通信行为
│
├── 情报置信度维度（Intelligence） 权重 15%
│   ├── 关联的恶意 IOC 数量和判定
│   ├── 相关 CVE 是否在野利用
│   └── 威胁情报来源可信度
│
└── 时效性维度（Urgency）         权重 10%
    ├── 事件首次发现到现在的时长
    └── 最近一次新告警加入的时间
```

#### 4.17.2 资产重要性等级

资产重要性是 SmartScore 影响维度的关键输入，按以下等级分类：

| 等级 | 资产类型 | 分值系数 |
|------|----------|----------|
| 极高 | 域控服务器、证书服务器、核心数据库 | 1.5× |
| 高 | 互联网暴露服务器、文件服务器、HR/财务系统 | 1.3× |
| 中 | 普通服务器、开发机 | 1.0× |
| 低 | 普通工作站 | 0.8× |
| 特殊 | 蜜罐（任何访问均视为高危）| 强制评分 ≥ 80 |

资产重要性等级由安全管理员在资产管理模块手动标注，或由系统根据资产类型自动推断（可覆盖）。

#### 4.17.3 评分动态更新触发条件

| 触发事件 | 说明 |
|----------|------|
| 新告警加入事件 | 重新计算全部维度 |
| 关联资产重要性变更 | 更新影响维度 |
| 威胁情报库更新命中关联 IOC | 更新情报维度 |
| 分析师手动调整 | 支持手动覆盖 SmartScore，记录原因和调整人 |
| 事件状态变为"已关闭" | 冻结分数，不再更新 |

#### 4.17.4 评分解释与透明度

- 事件详情页展示 SmartScore 得分条，悬停展示各维度贡献比例
- "为什么这么高？"按钮展示最高贡献因子的文字解释（可与 Copilot 联动）
- 历史分数折线图：展示事件生命周期内 SmartScore 的变化趋势
- 每次分数变更记录变更原因（新增告警/情报更新/手动调整）

#### 4.17.5 评分配置

- 安全管理员可调整各维度权重（总权重必须 = 100%）
- 可设置 SmartScore 阈值触发动作：
  - ≥ 80：自动推送通知给 L3 分析师
  - ≥ 90：自动推送给安全运营负责人
  - ≥ 90 且资产等级"极高"：自动触发 SOAR 紧急响应剧本

### 4.18 检测规则引擎

**功能描述：** 提供统一的安全检测规则管理平台，支持 BIOC（行为失陷指标）规则、IOC 匹配规则和统计异常规则的创建、测试、部署和版本管理。分析师可在平台内编写自定义规则，无需部署代码。

#### 4.18.1 规则类型

**BIOC 规则（行为失陷指标）：**

基于事件序列和进程行为的规则，描述攻击行为模式：

```
规则示例：
名称：Office 应用启动网络工具
触发条件：
  进程名 IN ['WINWORD.EXE', 'EXCEL.EXE', 'POWERPNT.EXE']
  AND 子进程名 IN ['powershell.exe', 'cmd.exe', 'mshta.exe', 'wscript.exe']
  AND 子进程包含命令行参数 '-enc' OR '-nop' OR 'IEX'
严重级别：High
MITRE 映射：T1566.001（钓鱼附件）, T1059.001（PowerShell）
```

**IOC 匹配规则：**

基于威胁情报的 IOC 实时匹配，当终端/网络事件的 IP/域名/哈希命中 IOC 库时触发告警：

- 支持匹配类型：IP、域名、URL、文件 SHA256/MD5、邮件地址
- 支持按 IOC 判定级别过滤（恶意/可疑/未知）
- 支持按情报来源白名单（只触发来自指定情报源的 IOC）

**统计异常规则（UEBA 规则）：**

基于基线行为统计，发现偏离正常模式的异常：

```
规则示例：
名称：用户单日文件访问量异常
统计对象：user_id
统计指标：文件访问事件数 per 日
基线方式：过去 30 天同用户同类型工作日的 P90 值
触发阈值：当日访问量 > 基线值的 5 倍
严重级别：Medium
MITRE 映射：T1039（网络共享数据采集）
```

#### 4.18.2 规则生命周期管理

| 状态 | 说明 |
|------|------|
| **草稿（Draft）** | 规则编写中，不生效 |
| **测试（Testing）** | 对历史数据运行回放，查看命中情况，不生成真实告警 |
| **已启用（Active）** | 生产环境实时运行 |
| **已禁用（Disabled）** | 保留规则但不运行 |
| **已弃用（Deprecated）** | 标记废弃，不可再启用 |

版本管理：
- 每次规则保存生成新版本（v1 / v2 / v3...）
- 支持查看历史版本内容和历史版本的命中统计
- 支持回滚到任意历史版本

#### 4.18.3 规则测试（历史回放）

- 在规则进入生产前，对过去 N 天的历史日志运行回放测试
- 回放结果展示：命中事件数、命中时间分布、命中的具体事件样本
- 支持估算误报率（与已知误报告警的重叠比例）
- 回放不产生真实告警，结果仅用于规则调优

#### 4.18.4 内置规则库

- 平台内置基于 MITRE ATT&CK 框架映射的检测规则集
- 按战术分类浏览（初始访问、执行、持久化、提权、防御规避...）
- 内置规则可克隆为自定义规则修改，不可直接编辑内置版本
- 定期更新内置规则库（版本迭代时推送新规则通知）

#### 4.18.5 规则命中统计

每条规则展示：
- 过去 7/30 天告警生成数
- 误报率（被标记为误报的占比）
- 平均处置时长
- 关联的事件数

低命中 + 高误报的规则系统标记"建议检查"，由安全管理员定期评审。

#### 4.18.6 MITRE ATT&CK 覆盖度看板

- 可视化矩阵展示当前启用规则对 MITRE ATT&CK 战术/技术的覆盖情况
- 绿色：已有规则覆盖；灰色：未覆盖；黄色：规则处于测试状态
- 点击具体技术格子，展示覆盖该技术的规则列表
- 支持导出覆盖度报告（PDF/JSON）

### 4.19 暴露管理（Exposure Management）

**功能描述：** 在传统漏洞管理（CVE 列表）基础上增加**可达性分析**维度，区分理论上存在的漏洞和在当前网络环境下实际可被攻击利用的漏洞，大幅减少运营团队处理的漏洞噪声，将有限精力集中在真正高风险的修复项上。

> **与 4.5 的关系**：4.5（脆弱性管理）负责 CVE 数据的 CRUD 和基础优先级。4.19 是在此基础上增加可达性分析层和修复编排能力，两者数据共享。

#### 4.19.1 可达性分析

**核心思路：** 一个 CVE 的实际威胁级别 = CVSS 基础分 × 在野利用系数 × 网络可达性系数 × 资产重要性系数。

可达性分析从以下维度评估漏洞是否"实际可被利用"：

| 维度 | 评估方法 | 说明 |
|------|----------|------|
| **网络可达性** | 分析防火墙策略和网络分区，判断攻击者能否从已知入口点到达漏洞资产 | 互联网直接暴露 > DMZ 内 > 内网隔离区 |
| **服务暴露面** | 漏洞对应的端口/服务是否处于开放监听状态 | 扫描数据与 CVE 关联的服务版本匹配 |
| **在野利用** | 威胁情报中是否有该 CVE 的活跃利用报告 | 来自威胁情报模块 |
| **补偿控制** | 是否已有其他安全控制缓解该漏洞（WAF 规则、网络 ACL、Agent 防护）| 可手动标注补偿控制降低可达性 |

#### 4.19.2 暴露优先级评分

综合以上维度，每条漏洞实例（CVE × 资产）计算**暴露优先级分**（Exposure Priority Score，0-100）：

```
Exposure Priority = 
  CVSS_Score × 10                          # 基础分（0-100）
  × InWildFactor（1.0 正常 / 1.5 在野利用）
  × ReachabilityFactor（0.3 隔离 / 0.7 内网 / 1.0 DMZ / 1.3 互联网暴露）
  × AssetImportanceFactor（0.8-1.5）
  × max(100)                               # 上限 100
```

> 高优先级（≥ 70）的漏洞是修复的核心关注对象，其余漏洞可进入等待队列。

#### 4.19.3 噪声过滤

- **过滤低可达性漏洞**：可达性因子 < 0.4 的漏洞默认折叠，不出现在主列表（可展开查看）
- **自动去重**：相同 CVE 影响多台同类型资产时，聚合展示（按资产组而非逐条展开）
- **过期 CVE 归档**：已修复超过 30 天的 CVE 自动归档，不占用主视图

目标：核心修复工作台展示的漏洞数量 ≤ 实际 CVE 总量的 20%（其余可在"完整列表"查看）。

#### 4.19.4 修复工作台

暴露管理提供专用的修复工作台视图，区别于 4.5 的 CVE 列表视图：

**视图组织：**
- 按"修复批次"组织（一个批次 = 一组可同时修复的 CVE，通常针对同一资产/同一补丁包）
- 每个批次展示：影响资产数、最高优先级分、预计修复工时、修复方式（补丁/配置变更/隔离）

**修复状态跟踪：**

| 状态 | 说明 |
|------|------|
| 未计划 | 新发现，尚未分配修复责任人 |
| 已计划 | 已分配责任人和修复截止日期 |
| 修复中 | 已开始修复（补丁下载/测试阶段） |
| 验证中 | 声称已修复，等待系统重新扫描确认 |
| 已修复 | 扫描确认漏洞消失 |
| 接受风险 | 决策者确认接受该漏洞风险（需填写原因和有效期） |
| 有补偿控制 | 已部署缓解措施，暂不修复（需填写控制措施描述） |

#### 4.19.5 修复编排与 SOAR 联动

高优先级漏洞修复可触发 SOAR 剧本自动化部分修复流程：

- **自动创建工单**：在 ITSM（Jira/ServiceNow）自动创建修复工单，附带 CVE 详情、受影响资产清单、修复建议
- **通知责任人**：自动发送修复通知给资产责任人（邮件/企微/钉钉）
- **截止日期提醒**：修复截止日期前 3 天/1 天自动发送提醒
- **扫描触发**：修复声明后自动触发重新扫描任务，验证修复结果

联动通过 SOAR 的"漏洞自动工单"模板剧本实现（见 5.3.5），XSIAM 通过 Webhook 推送 `vulnerability.discovered` 事件到 SOAR。

#### 4.19.6 暴露面统计看板

- **修复进度仪表盘**：高/极高优先级漏洞的修复率（已修复/总数），带趋势
- **暴露面趋势**：过去 30 天每日高优先级暴露漏洞数量变化曲线
- **Top 10 高风险漏洞**：按暴露优先级分排序的未修复漏洞列表
- **按资产类型分布**：服务器/终端/网络设备的高优先级漏洞占比
- **SLA 达成情况**：Critical CVE 要求 7 天内修复的达成率

### 4.20 身份威胁检测与响应（ITDR）

**功能描述：** 在 4.12（身份安全/UEBA）的基础上，增加**实时身份风险引擎**和**动态权限收缩**能力，实现对身份攻击的主动防御。ITDR 不仅检测已发生的身份异常，还能在攻击进行中实时限制受损账号的权限范围，降低横向移动和权限提升的影响范围。

> **与 4.12 的关系**：4.12 描述身份安全的检测场景和 UEBA 分析；4.20 描述实时风险引擎架构、动态权限收缩机制和特权账号全生命周期管理，是 4.12 的深化实现规格。

#### 4.20.1 实时身份风险引擎

**用户风险评分（0-100）持续计算，而非批量快照更新：**

| 风险信号 | 权重 | 说明 |
|----------|------|------|
| 地理位置异常（新国家/不可能旅行）| 高（+30） | 同账号在不可能的时间从两地登录 |
| 登录时间异常（非工作时间、节假日）| 中（+15） | 基于用户历史行为基线 |
| 新设备登录 | 中（+15） | 从未见过的设备指纹 |
| 认证失败率 | 高（+25 escalating） | 失败次数越多分值越高 |
| 敏感资源首次访问 | 中（+20） | 首次访问高价值数据/系统 |
| 权限使用异常 | 高（+25） | 使用从未用过的高权限操作 |
| 关联的活跃告警 | 高（×1.2 乘数） | 当前有活跃安全告警关联该账号 |
| 关联的活跃事件 | 极高（×1.5 乘数）| 账号已被纳入活跃安全事件 |

风险分变更时：
- 实时更新用户详情页和告警列表中的风险标签
- 超过阈值（默认 70）自动创建 ITDR 告警
- 超过阈值（默认 85）触发动态权限收缩（见 4.20.2）

#### 4.20.2 动态权限收缩（Dynamic Privilege Restriction）

**核心能力：** 当用户风险分超过配置阈值时，系统自动（或由分析师手动）对该账号在目标 IdP 系统中施加临时限制，不需要完全禁用账号：

| 收缩级别 | 触发条件 | 限制内容 | 说明 |
|----------|----------|----------|------|
| **L1 监控强化** | 风险分 ≥ 70 | 登录行为全量审计，异常立即告警 | 无感知，不影响用户操作 |
| **L2 访问限制** | 风险分 ≥ 80 | 禁止访问高价值资源（可配置资源组）| 用户仍可正常工作，仅限敏感操作 |
| **L3 强制 MFA** | 风险分 ≥ 85 | 每次敏感操作触发 MFA 二次验证 | 对攻击者无效，对正常用户有轻微摩擦 |
| **L4 会话撤销** | 风险分 ≥ 90 | 强制注销所有活跃会话，下次登录需 MFA | 中断当前连接，需用户重新认证 |
| **L5 账号暂停** | 风险分 ≥ 95 或人工判断 | 在 AD/IdP 中暂停账号 | 完全阻断，需管理员恢复 |

**限制执行方式：**
- 通过 SOAR 的 Identity Adapter（AD/Azure AD/Okta）执行
- ITDR 模块向 SOAR 发送触发信号，SOAR 调用对应 Adapter
- 每次收缩动作记录触发原因、执行时间、执行人（自动/手动）、恢复时间

**自动恢复机制：**
- 收缩动作设置自动过期时间（L1/L2：24h，L3：8h，L4/L5：需人工确认恢复）
- 风险分下降到阈值以下后，系统提醒分析师评估是否解除限制
- 解除记录纳入审计日志

#### 4.20.3 特权账号全生命周期管理

**特权账号识别：**

系统自动识别以下类型特权账号（可手动补充）：
- Domain Admin / Enterprise Admin
- 本地 Administrator
- 数据库管理员账号（DBA）
- 云平台根账号/高权限 IAM 角色
- 服务账号（Service Account）

**全生命周期状态追踪：**

| 阶段 | 说明 |
|------|------|
| 创建 | 记录创建时间、申请人、审批人、用途说明 |
| 活跃使用 | 追踪每次使用的时间、来源 IP、操作内容、持续时长 |
| 异常使用 | 非常规时间/地点/操作触发 ITDR 告警 |
| 长期未使用 | 超过 90 天未使用的特权账号标记"僵尸账号"，发起审查流程 |
| 注销 | 记录注销原因和执行人 |

**特权账号看板：**
- 所有特权账号列表（按账号类型分组）
- 每个账号的风险评分和最后使用时间
- "僵尸账号"列表（超过 N 天未用，可配置）
- 本周特权操作次数趋势
- 特权账号使用异常告警列表

#### 4.20.4 身份攻击场景检测增强

在 4.12 的检测场景基础上，ITDR 增加以下检测能力：

| 检测场景 | 说明 | 对应 MITRE |
|----------|------|------------|
| 票据传递（Pass-the-Ticket）| 异常 Kerberos 票据使用，票据寿命异常长 | T1550.003 |
| AS-REP Roasting | 无预认证的账号被枚举，可能被离线破解 | T1558.004 |
| Kerberoasting | SPN 账号票据被请求导出，可能被离线破解 | T1558.003 |
| LDAP 侦察 | 大量 LDAP 查询枚举 AD 对象 | T1087.002 |
| AdminSDHolder 滥用 | 受保护 AD 对象的 ACL 被异常修改 | T1484 |
| Shadow Credentials | msDS-KeyCredentialLink 属性被恶意写入 | T1556 |
| MFA 疲劳攻击 | 短时间内大量 MFA 推送请求同一账号 | T1621 |

#### 4.20.5 身份风险与事件联动

- ITDR 生成的告警自动参与 CAE 的因果关联，可与终端告警、网络告警合并为统一事件
- 事件详情页增加"身份视角"标签：展示事件涉及账号的完整风险时间线
- 高风险身份告警（风险分 ≥ 85）自动在事件 SmartScore 计算中触发×1.5 乘数

### 4.21 多租户与权限管理

**功能描述：** 支持多租户（Multi-Tenant）隔离架构，满足大型集团企业（多业务单元隔离）和 MSSP（托管安全服务提供商）的管理需求。在单一平台部署的基础上，不同租户的数据、配置和操作完全隔离，同时支持上层管理视图跨租户聚合查看。

#### 4.21.1 租户模型

**三层租户结构：**

```
超级租户（Super Tenant）
└── 管理员可以查看所有子租户的聚合态势
    ├── 子租户 A（如：集团总部）
    │   ├── 独立的资产、告警、事件数据
    │   ├── 独立的检测规则和 SOAR 剧本
    │   └── 独立的用户和角色
    ├── 子租户 B（如：子公司 / 客户 A）
    └── 子租户 C（如：子公司 / 客户 B）
```

**数据隔离保证：**
- 每个租户的所有数据打上 `tenant_id` 标签，API 层强制过滤
- 子租户用户无法跨租户查询数据
- 超级租户管理员可切换视图进入任意子租户上下文

#### 4.21.2 超级租户管理控制台

超级租户管理员拥有跨租户聚合视图：

| 功能 | 说明 |
|------|------|
| 全局态势聚合 | 所有子租户的事件数、告警数、高风险资产数的汇总 |
| 子租户健康监控 | 每个子租户的 Agent 在线率、数据接入状态、最后告警时间 |
| 租户切换 | 一键切换进入任意子租户的完整操作上下文 |
| 租户创建/注销 | 新建子租户（初始化默认配置）、注销租户（数据保留策略） |
| 跨租户规则推送 | 将检测规则/剧本模板推送到一个或多个子租户 |
| 全局用户管理 | 管理跨租户的用户账号（如 MSSP 分析师可服务多个客户） |

#### 4.21.3 对象级 RBAC（Object-Level Access Control）

在原有 6 角色 RBAC 基础上，增加对象级权限控制：

**对象级权限支持的资源类型：**

| 资源类型 | 可配置权限粒度 |
|----------|---------------|
| 检测规则 | 只读 / 可编辑 / 可部署 / 可删除 |
| SOAR 剧本 | 只读 / 可编辑 / 可触发 / 可管理 |
| 报表模板 | 只读 / 可生成 / 可编辑 / 可删除 |
| 资产组 | 只读 / 可编辑 / 可执行响应动作 |
| 日志数据集 | 只读 / 可查询（按数据集类型限制） |
| 威胁情报源 | 只读 / 可配置 / 可同步 |

**配置方式：** 管理员在"角色管理"页面，可以为指定用户组追加或收窄对特定对象集合的权限，覆盖其基础角色权限（只能收窄，不能超出角色基础权限上限）。

#### 4.21.4 数据隔离与审计

- **API 层强制隔离**：所有数据读写 API 在 Handler 层注入 `tenant_id` 过滤，框架级别保证不泄露
- **跨租户操作审计**：超级租户管理员的跨租户操作（切换、推送规则、查看数据）完整记录在超级租户审计日志
- **数据导出隔离**：导出接口只能导出当前 `tenant_id` 范围内的数据，任何试图跨租户导出的请求被记录并拒绝

#### 4.21.5 MSSP 场景支持

针对 MSSP（托管安全服务提供商）的专项支持：

- **分析师跨租户服务**：MSSP 分析师账号可被授权服务多个客户租户，在不同租户之间切换无需重新登录
- **白标支持（White-label）**：子租户可配置独立的 Logo 和平台名称（UI 层替换）
- **客户自助视图**：子租户客户可开通只读观察员账号，自行查看态势和报表，无需通过 MSSP
- **SLA 追踪**：超级租户控制台展示每个子租户的响应 SLA 达成率，支持按客户导出 SLA 报告

## 5. SOAR 子系统

### 5.1 SOAR 定位与边界

**功能描述：** SOAR 是独立于 XSIAM 的安全编排自动化引擎，负责将安全响应流程标准化、自动化，并通过可插拔的 Action Adapter 层调用各类安全设备执行真实处置动作。

**SOAR 职责范围：**

| 职责 | 说明 |
|------|------|
| 剧本编排 | 可视化编排响应流程（条件/分支/并行/审批/子剧本） |
| 触发管理 | 接收 XSIAM Webhook、定时、手动、API 多种触发信号 |
| 设备动作执行 | 通过 Action Adapter 调用真实设备 API 执行处置 |
| 执行状态回写 | 执行结果异步回写到 XSIAM Action Center |
| 审计可观测 | 每次执行的完整链路日志、耗时、成功/失败详情 |

**SOAR 不做：**
- 不做威胁检测和告警分析（XSIAM 的职责）
- 不存储原始日志和遥测数据（XSIAM 的职责）
- 不直接展示 XSIAM 的告警/事件详情（通过 API 查询 XSIAM）

---

### 5.2 XSIAM ↔ SOAR 集成协议

#### 5.2.1 XSIAM → SOAR：事件推送（Webhook）

XSIAM 在满足条件时向 SOAR 推送事件，触发剧本执行。

**Webhook 事件 Schema：**

```json
{
  "event_id":    "EVT-20260522-001",
  "event_type":  "incident.created | incident.severity_changed | alert.triggered | ioc.matched",
  "occurred_at": "2026-05-22T09:41:02Z",
  "source":      "xsiam",
  "payload": {
    "incident_id":   "INC-2026-0042",
    "severity":      "critical",
    "smart_score":   87,
    "alert_count":   12,
    "affected_assets": ["asset-001", "asset-002"],
    "mitre_tactics": ["TA0003", "TA0010"],
    "assignee_id":   "user-123"
  },
  "meta": {
    "xdr_version": "1.0",
    "tenant_id":   "tenant-abc"
  }
}
```

**推送触发条件（可配置）：**

| 事件类型 | 触发条件 |
|----------|----------|
| `incident.created` | 新事件创建（可按严重级别过滤） |
| `incident.severity_changed` | 事件严重级别上升（如 Medium → Critical） |
| `alert.triggered` | 告警匹配指定检测规则时 |
| `ioc.matched` | IOC 命中（指定 IOC 类型和判定级别） |
| `vulnerability.discovered` | 新严重漏洞发现（CVSS ≥ 9.0）|

**Webhook 可靠性要求：**
- XSIAM 推送失败时指数退避重试（最大 3 次，间隔 1s/5s/30s）
- SOAR 接收后返回 200 确认，XSIAM 记录推送状态
- 支持 SOAR 主动拉取（XSIAM 提供 `GET /api/events?since=` 接口，作为 Webhook 的备份机制）

#### 5.2.2 SOAR → XSIAM：状态回写（REST API）

SOAR 执行完毕后通过 XSIAM API 回写结果：

| API | 说明 |
|-----|------|
| `PATCH /api/actions/:id` | 回写动作执行状态（running / completed / failed） |
| `POST /api/incidents/:id/notes` | 将执行摘要写入事件备注 |
| `POST /api/audit-logs` | 写入 SOAR 来源的审计记录 |

**回写数据格式：**
```json
{
  "status":       "completed",
  "result_summary": "终端 WIN-001 已成功隔离，网络连接已切断",
  "result_detail": {
    "adapter":     "endpoint-agent-v2",
    "execution_id": "EXEC-20260522-00123",
    "duration_ms":  1240,
    "output":      "Isolation policy applied. Agent heartbeat maintained."
  },
  "completed_at": "2026-05-22T09:41:03Z"
}
```

#### 5.2.3 SOAR → XSIAM：数据查询（REST API）

剧本中的查询节点可调用 XSIAM API 获取上下文数据：

| API | 说明 |
|-----|------|
| `GET /api/assets/:id` | 获取资产详情（判断是否为关键资产） |
| `GET /api/alerts?incident_id=` | 获取事件关联告警 |
| `GET /api/threat-intel/iocs/search` | 查询 IOC 情报 |
| `POST /api/logs/query` | 执行 XQL 查询（数据湖 Stub） |

### 5.3 剧本管理（Playbook）

**功能描述：** 可视化剧本编排工具，将安全响应流程以节点图方式定义，由 SOAR 执行引擎运行。

#### 5.3.1 触发方式

| 触发方式 | 说明 |
|----------|------|
| **XSIAM Webhook 触发** | 收到 XSIAM 推送的事件满足剧本触发条件时自动启动 |
| **定时触发（Cron）** | 按 Cron 表达式定时执行（如每天 9:00 执行威胁巡检剧本） |
| **手动触发** | 分析师在 XSIAM 事件详情页点击"执行剧本"按钮，SOAR 接收请求启动 |
| **API 触发** | 外部系统通过 `POST /soar/api/playbooks/:id/trigger` 触发 |
| **IOC 规则触发** | IOC 命中规则后由 SOAR 内部触发（不经过 XSIAM） |

#### 5.3.2 剧本节点类型

| 节点类型 | 节点标识 | 说明 |
|----------|---------|------|
| **触发器** | `trigger` | 剧本入口，定义触发条件和入参 |
| **条件判断** | `condition` | IF/ELSE 逻辑分支，基于上下文变量判断路径 |
| **Action 节点** | `action` | 调用指定 Adapter 执行设备动作（见 5.4） |
| **查询节点** | `query` | 调用 XSIAM API 查询数据，结果传递给下游节点 |
| **通知节点** | `notify` | 发送邮件、钉钉/企微、Slack、Webhook 通知 |
| **审批节点** | `approval` | 暂停执行，等待指定人员审批后继续或终止 |
| **延时节点** | `delay` | 等待指定时长后继续（最大 24 小时） |
| **脚本节点** | `script` | 执行 Python/Go 内联脚本，处理复杂逻辑 |
| **子剧本** | `sub_playbook` | 调用另一个剧本作为子流程，支持参数传递 |
| **并行网关** | `parallel` | 同时触发多个分支，等待全部完成后合并 |
| **ITSM 节点** | `itsm` | 在 ITSM 系统（如 ServiceNow、Jira）创建/更新工单 |
| **结束节点** | `end` | 剧本正常结束，可配置回写 XSIAM 的摘要内容 |

#### 5.3.3 剧本上下文变量

剧本执行时维护上下文变量（Context），节点间可引用：

```
{{ trigger.incident_id }}        XSIAM 推送的事件 ID
{{ trigger.severity }}           严重级别
{{ trigger.affected_assets[0] }} 第一个受影响资产 ID
{{ query.asset_detail.type }}    查询节点返回的资产类型
{{ action.isolate.result }}      隔离动作的执行结果
{{ approval.reviewer }}          审批人
```

#### 5.3.4 剧本管理功能

- 可视化拖拽式剧本编辑器（节点画布，React Flow 实现）
- 剧本版本管理（查看历史版本、对比差异、回滚）
- **测试模式（Dry Run）**：不执行 Action 节点，仅模拟流程路径，输出预期执行计划
- 剧本克隆（基于模板快速创建）
- 剧本启用/禁用（不删除，保留执行历史）
- 剧本导入/导出（JSON 格式，支持跨环境迁移）

#### 5.3.5 内置剧本模板

| 模板名称 | 触发条件 | 主要流程 |
|----------|----------|----------|
| 勒索软件响应 | incident.created，severity=critical | 隔离终端 → 取消网络访问 → 收集取证 → 通知安全团队 → 创建 ITSM 工单 |
| 钓鱼邮件处置 | alert.triggered，rule=phishing | 查询发件人 IOC → 封锁域名 → 禁用点击用户账号 → 通知用户 |
| 账号攻击响应 | alert.triggered，rule=brute_force | 判断账号类型 → 特权账号强制禁用 → 普通账号强制 MFA → 通知 |
| IOC 封锁工作流 | ioc.matched，verdict=malicious | 查询内网命中资产 → 封锁 IP/域名 → 扫描关联终端 → 记录到事件 |
| 漏洞自动工单 | vulnerability.discovered，cvss≥9 | 查询受影响资产责任人 → 创建 Jira 工单 → 设置修复截止日期 → 通知 |
| 端点隔离工作流 | 手动触发 | 审批确认 → 隔离终端 → 启动实时终端取证 → 回写 XSIAM |

### 5.4 Action Adapter 层（设备执行接口）

**功能描述：** Action Adapter 是 SOAR 与各类安全设备之间的标准化适配层，每种设备类型封装为独立 Adapter，通过统一接口注册到 SOAR 执行引擎。

#### 5.4.1 Adapter 设计原则

| 原则 | 说明 |
|------|------|
| **统一接口** | 所有 Adapter 实现相同的 `Execute(action_type, params) → Result` 接口 |
| **独立注册** | 每个 Adapter 作为独立插件注册，新增 Adapter 不需修改执行引擎代码 |
| **配置驱动** | Adapter 的连接参数（地址、凭证、超时）通过配置管理，不硬编码 |
| **幂等设计** | 同一动作重复执行结果一致，避免副作用叠加 |
| **超时隔离** | 每个 Adapter 调用有独立超时，单个 Adapter 超时不阻塞整个剧本 |
| **错误标准化** | 所有 Adapter 错误统一转换为标准错误码（`ADAPTER_TIMEOUT` / `AUTH_FAILED` / `TARGET_NOT_FOUND` 等） |

#### 5.4.2 内置 Adapter 类型

**终端 Adapter（Endpoint Adapters）：**

| Adapter | 支持动作 | 对接方式 |
|---------|----------|----------|
| `xsiam-agent` | isolate / unisolate / kill_process / quarantine_file / run_script / collect_forensics | XSIAM Agent REST API |
| `crowdstrike` | isolate / unisolate / run_script | CrowdStrike RTR API |
| `sentinelone` | isolate / unisolate / kill_process | SentinelOne API |

**网络设备 Adapter（Network Adapters）：**

| Adapter | 支持动作 | 对接方式 |
|---------|----------|----------|
| `paloalto-ngfw` | block_ip / block_url / block_domain | PAN-OS REST API |
| `fortinet` | block_ip / block_domain | FortiOS REST API |
| `cisco-asa` | block_ip | Cisco ASA REST API |
| `dns-blackhole` | block_domain | 内部 DNS 服务 API |

**身份 Adapter（Identity Adapters）：**

| Adapter | 支持动作 | 对接方式 |
|---------|----------|----------|
| `active-directory` | disable_account / reset_password / enforce_mfa | LDAP / AD REST API |
| `azure-ad` | disable_account / revoke_sessions / enforce_mfa | Microsoft Graph API |
| `okta` | disable_account / revoke_sessions / reset_password | Okta API |

**通知 Adapter（Notification Adapters）：**

| Adapter | 支持动作 | 对接方式 |
|---------|----------|----------|
| `email` | send_email | SMTP |
| `dingtalk` | send_message | 钉钉 Webhook |
| `wecom` | send_message | 企业微信机器人 Webhook |
| `slack` | send_message | Slack Incoming Webhook |
| `pagerduty` | create_incident / resolve_incident | PagerDuty API |

**ITSM Adapter：**

| Adapter | 支持动作 | 对接方式 |
|---------|----------|----------|
| `servicenow` | create_ticket / update_ticket / close_ticket | ServiceNow REST API |
| `jira` | create_issue / update_issue | Jira REST API |

#### 5.4.3 Adapter 注册与管理界面

SOAR 提供 Adapter 管理 UI：
- 查看所有已注册 Adapter（名称、类型、版本、健康状态）
- 配置 Adapter 连接参数（地址、API Key、超时、重试次数）
- 对每个 Adapter 执行连接测试（Test Connection）
- 查看 Adapter 最近执行记录（成功率、平均耗时）
- 启用/禁用指定 Adapter

#### 5.4.4 Adapter 统一接口规范

```
接口：Execute(ctx, actionType, params) → (Result, error)

输入：
  actionType  string          动作类型（如 "endpoint.isolate"）
  params      map[string]any  动作参数（如 {"asset_id": "...", "reason": "..."}）

输出：
  Result.Success       bool    是否成功
  Result.ExecutionID   string  本次执行的唯一 ID（用于追踪）
  Result.Message       string  执行摘要（人类可读）
  Result.Detail        any     执行详情（结构化，适配器自定义）
  Result.Duration      int     执行耗时（毫秒）

错误码：
  ADAPTER_TIMEOUT       适配器调用超时
  AUTH_FAILED           认证失败（API Key 无效）
  TARGET_NOT_FOUND      目标资产/账号不存在
  ACTION_NOT_SUPPORTED  该适配器不支持该动作类型
  DEVICE_UNAVAILABLE    目标设备不可达
  PERMISSION_DENIED     权限不足
```

### 5.5 执行引擎

**功能描述：** SOAR 的核心运行时，负责解析剧本 DAG（有向无环图），按节点类型调度执行，处理条件分支、并行、审批等复杂流程控制。

#### 5.5.1 执行流程

```
1. 接收触发信号（Webhook / Cron / 手动 / API）
   │
2. 匹配剧本（按触发类型 + 触发条件过滤，找到所有匹配的启用剧本）
   │
3. 创建执行实例（Execution Instance），初始化 Context
   │
4. 按 DAG 顺序遍历节点：
   ├── condition  → 评估表达式，选择下一节点
   ├── action     → 调用对应 Adapter.Execute()
   ├── query      → 调用 XSIAM API，结果写入 Context
   ├── notify     → 调用通知 Adapter
   ├── approval   → 挂起实例，等待人工审批
   ├── delay      → 调度器延时后继续
   ├── parallel   → Fork 多个子线程，Barrier 等待全部完成
   └── sub_playbook → 递归启动子剧本实例
   │
5. 到达 end 节点：
   ├── 汇总执行结果
   ├── 异步回写 XSIAM（PATCH /api/actions/:id）
   └── 写入执行审计日志
```

#### 5.5.2 执行控制

| 特性 | 说明 |
|------|------|
| **最大执行时长** | 单次剧本执行最长 2 小时（含审批等待），超时自动终止并告警 |
| **节点超时** | 每个 Action 节点独立超时配置（默认 30 秒，可配置 1-300 秒） |
| **自动重试** | Action 节点失败后可配置重试次数（默认 0，最大 3 次），指数退避 |
| **并发限制** | 同一剧本最多同时运行 N 个实例（默认 10，可配置） |
| **人工介入** | 分析师可随时暂停/继续/终止正在执行的实例 |
| **Dry Run 模式** | 测试模式下跳过所有 Action 节点，仅演算执行路径 |

### 5.6 执行审计与可观测性

**功能描述：** 记录每次剧本执行的完整生命周期，提供可查询的执行历史和性能分析。

#### 5.6.1 执行实例记录

每次执行实例记录：

| 字段 | 说明 |
|------|------|
| 执行 ID | 唯一标识（EXEC-YYYYMMDD-XXXXX） |
| 剧本名称 / 版本 | 执行的剧本及版本号 |
| 触发方式 | xdr_webhook / cron / manual / api |
| 触发源数据 | XSIAM 推送的原始 Webhook payload |
| 关联 XSIAM 事件 | incident_id / alert_id（如有） |
| 执行状态 | running / completed / failed / cancelled / timed_out |
| 开始时间 / 结束时间 | UTC 时间戳 |
| 总耗时 | 毫秒 |
| 节点执行日志 | 每个节点的输入/输出/耗时/状态 |
| 最终结果摘要 | 人类可读的执行总结 |

#### 5.6.2 节点级执行日志

每个节点记录：
- 节点类型和名称
- 输入参数（Context 快照）
- 输出结果
- 执行耗时
- 状态（success / failed / skipped / pending_approval）
- 错误信息（如失败）
- 使用的 Adapter 和版本（Action 节点）

#### 5.6.3 执行审计查询

- 按剧本名称、触发方式、执行状态、时间范围过滤
- 按关联 XSIAM 事件 ID 查询（从 XSIAM 跳转到 SOAR 执行历史）
- 支持导出执行报告（CSV / JSON）

## 8. 非功能性需求

### 8.1 性能要求

| 指标 | 要求 |
|------|------|
| 页面加载时间 | 主要功能页面首次加载 ≤ 3 秒 |
| 告警入库延迟 | 告警从产生到在平台可见 ≤ 60 秒 |
| XQL 查询响应 | 常规查询（≤ 24h、单数据集）≤ 5 秒 |
| 实时终端延迟 | 命令发送到响应 ≤ 500ms |
| 并发用户数 | 支持 200 并发用户无性能降级 |
| 告警处理吞吐量 | 支持每天处理 1000 万+ 原始事件 |

### 8.2 可靠性要求

| 指标 | 要求 |
|------|------|
| 服务可用性（SLA） | ≥ 99.9%（月度） |
| RPO（恢复点目标） | ≤ 1 小时 |
| RTO（恢复时间目标） | ≤ 4 小时 |
| 数据备份 | 每日全量备份 + 实时增量 |

### 8.3 安全要求

- 全量 HTTPS（TLS 1.2+）通信加密
- Agent 通信使用双向 TLS 认证
- 所有管理操作记录审计日志（不可删除）
- 支持 SSO（SAML 2.0 / OIDC）集成
- 支持 MFA（多因素认证）强制启用
- 敏感操作（删除数据/高危响应）需双人授权
- 数据落盘加密（AES-256）

### 8.4 合规要求

- 日志保留满足等保 2.0 三级要求（≥ 6 个月在线，≥ 1 年归档）
- 支持导出合规审计报告
- 操作审计日志不可篡改

### 8.5 浏览器兼容性

- Chrome ≥ 90
- Firefox ≥ 88
- Edge ≥ 90
- 不支持 IE 系列
